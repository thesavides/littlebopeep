## 1. What This Project Is

Little Bo Peep is a web platform connecting countryside walkers who spot lost or straying livestock with the farmers who need to recover them. Walkers submit geo-tagged reports with photos. Farmers claim and resolve reports. Admins manage the whole platform.

## 6. Critical Implementation Patterns

### API routes must NOT use `export const runtime = 'edge'`
The `open-next.config.ts` uses `wrapper: "cloudflare-node"` for the default handler. Routes tagged `export const runtime = 'edge'` are compiled into the **pure edge runtime** (no Node.js compat), which breaks `@supabase/supabase-js` (throws `TypeError: Cannot read properties of undefined (reading 'default')` at runtime). The working `delete-user` route never had this annotation — it is the reference pattern. Do not add `export const runtime = 'edge'` to any API route.

### `supabase.auth.admin` cannot be called client-side
Admin methods (`deleteUser`, `getUserById`, `listUsers`) require the service role key, which only exists server-side (Cloudflare Worker secret). Always use a dedicated `/api/admin/` route for these operations. The `deleteUser` function in `unified-auth.ts` is the reference pattern — it calls `/api/admin/delete-user` with a Bearer token.

### `suspendUser` / `activateUser` use the Supabase anon client (RLS-permitted)
`unified-auth.ts` `suspendUser()` and `activateUser()` call `updateUserProfile()` which uses the anon client with the user's active session. This works because Supabase RLS policies allow admins to update other users' profiles. After calling these, always refresh `realUsers` via `getAllUsers()` so the table re-renders.

### Zustand store vs Supabase — always call both
For any user state mutation (suspend, activate, role change, delete), always:
1. Call the `unified-auth` library function to persist to Supabase
2. Refresh `realUsers` via `getAllUsers()` so tables update
3. Optionally call the Zustand store action for optimistic UI

The store actions (`suspendUser`, `activateUser`, `deleteUser`) only update local in-memory state. The real users table is driven by `realUsers` (from Supabase), not the Zustand `users` array.

### Farmer dropdown includes admins
Both `CreateFarmModal` and `ClaimReportModal` receive `[...farmers, ...admins]` as the owners list:
```typescript
farmers={[...farmers, ...admins]}
```
When rendering names: `farmer.full_name || farmer.name || farmer.email`
(`full_name` = Supabase UserProfile field, `name` = legacy Zustand mock field)

### React-Leaflet map centre is NOT reactive
`MapContainer`'s `center` prop only sets the initial position. To move the map programmatically, use the `FlyToLocation` inner component in `MapInner.tsx`:
```typescript
function FlyToLocation({ center, zoom }: { center: [number, number]; zoom: number }) {
  const map = useMap()
  useEffect(() => {
    map.flyTo(center, zoom, { animate: true, duration: 1.2 })
  }, [center[0], center[1], zoom])
  return null
}
```

### Map marker types
`Map.tsx` (wrapper) and `MapInner.tsx` must have **identical** marker type unions:
```typescript
type?: 'default' | 'fencepost' | 'sheep' | 'existing' | 'selected' | 'user-location' | 'walker-location' | 'farmer-location'
```
If you add a new type in `MapInner.tsx`, add it to `Map.tsx` too or the TypeScript build will fail.

### `report_categories` are stored in two places
1. Supabase `report_categories` table (authoritative)
2. Zustand store (persisted to localStorage via `persist` middleware)

Categories are available **offline** without extra work — the offline capture form reads from the store.

### `condition` field is a free string
Migration 006 dropped the old `CHECK` constraint on `sheep_reports.condition`. It is now a plain `TEXT` field. Do not add a new CHECK constraint.

### Category subscription mode
`ReportCategory` has a `subscriptionMode: 'compulsory' | 'default_on' | 'default_off'` field (set in admin category editor). Farmer notification defaults are computed by `getEffectiveSubscription(farm, categoryId, categories)` in `appStore.ts`. The Supabase `report_categories` table does **not** yet have a `subscription_mode` column — this is stored only in the Zustand store/localStorage for now.
